# 카카오톡 물류 주문 자동화 시스템 - 설계 문서

> 작성일: 2026-03-06 | 상태: 승인됨

---

## 1. 프로젝트 개요

### 1.1 문제

물류 업체에서 화주(고객)들이 카카오톡 단톡방/1:1방으로 배송 주문을 보낸다.
메시지는 비정형 텍스트와 이미지(운송장, 물품 사진 등)로 들어오며,
현재는 사람이 직접 읽고 배차 앱(24시 전국 특송, 인성 퀵 등)에 수동 입력하고 있다.

### 1.2 목표

카톡 대화를 실시간으로 캡처 → AI가 정형화 → 사용자 확인/수정 → 배차 앱으로 자동 전송하는 **Electron 데스크톱 앱**을 만든다.

### 1.3 핵심 요구사항

| 항목 | 내용 |
|------|------|
| 카톡 캡처 | PC 카톡 20개+ 채팅방 실시간 모니터링 |
| 데이터 추출 | 텍스트 메시지 + 이미지(OCR) |
| AI 필터링 | 물류 주문 관련 메시지만 선별 (2단계 필터링) |
| AI 파싱 | 비정형 메시지 → 출발지, 도착지, 물품, 금액 등 JSON 정형화 |
| 사용자 확인 | 파싱 결과를 UI에서 확인/수정 |
| 배차 전송 | 배차 앱(24시 전국 특송, 인성 퀵 등)으로 API/웹자동화 전송 |
| 앱 형태 | Electron 데스크톱 앱 (모니터 + 프론트엔드 통합) |

### 2.1 기술 스택

| 계층 | 기술 | 선정 사유 |
|------|------|-----------|
| 데스크톱 앱 | **Electron** | 프론트엔드 + 백그라운드 프로세스 통합 패키징 |
| 프론트엔드 | **React Native Web** | 추후 모바일 확장 가능성, React 생태계 활용 |
| 카톡 모니터 | **Python** (pywinauto/uiautomation) | Windows UI Automation 라이브러리 성숙도 |
| AI 분류 | **Claude Haiku** | 빠르고 저렴한 YES/NO 분류 |
| AI 파싱 | **Claude Sonnet** | 한글 정확도 높음, 정밀 파싱 |
| AI OCR | **Claude Vision** | 이미지 내 한글 텍스트 인식 정확도 |
| DB | **SQLite** | 로컬 앱에 적합, 설치 불필요 |
| 배차 연동 | **Playwright** (웹 자동화) / REST API | 배차 앱 상황에 따라 선택 |

## 3. KakaoTalk Monitor 설계

### 3.1 동작 방식

```
[시작] → 카톡 PC 앱 윈도우 탐색
          |
          v
    설정 파일에서 모니터링 대상 채팅방 목록 로드
          |
          v
    +--- 루프 (2~3초 간격) ---+
    |                         |
    |  방 1 → 마지막 메시지 해시 비교 → 새 메시지 감지 시 캡처
    |  방 2 → ...
    |  ...
    |  방 N → ...
    |                         |
    +----- 한 바퀴: ~30~60초 --+
          |
    새 메시지 감지 시
          |
    +-----+------+
    | 텍스트      | 이미지
    | → 바로 전송  | → 캐시 폴더에서 파일 추출
    |            | → Claude Vision으로 OCR
    +-----+------+
          |
          v
    Electron Main Process로 IPC 전송
```

### 3.2 핵심 구현 사항

| 항목 | 구현 방식 |
|------|-----------|
| 윈도우 탐색 | `pywinauto` 또는 `uiautomation`으로 카톡 창 핸들 확보 |
| 텍스트 추출 | UI Automation의 Text 속성, 리스트 아이템 순회 |
| 새 메시지 감지 | 각 방의 마지막 메시지 해시값 저장 후 비교 |
| 이미지 추출 | `%LOCALAPPDATA%\Kakao\KakaoTalk\users\{hash}\Chats\` 캐시 폴더 감시 |
| 방 전환 | 채팅 목록에서 프로그래밍적으로 방 선택 → 대화 읽기 → 다음 방 |
| Electron 연동 | Python child_process로 실행, stdout/IPC로 데이터 전달 |

### 3.3 운영 권장사항

- **모니터링 전용 PC/카톡 계정 권장**: 방 전환 시 클릭 이벤트 발생으로 사용자 직접 사용과 충돌 가능
- 카톡 창은 최소화 상태에서도 UI Automation 동작 가능 (일부 제약)

## 4. AI 처리 설계 (2단계 필터링 + 파싱)

### 4.1 1단계: 분류 (Claude Haiku)

목적: 물류 주문 메시지인지 아닌지 빠르게 판별

```
입력: 카톡 메시지 (텍스트 또는 OCR 결과)
출력: YES / NO

분류 기준:
- 출발지/도착지가 포함된 배송 요청 → YES
- 물품 운송 관련 내용 → YES
- 운송장/송장/주문서 이미지 → YES
- 일상 대화, 이모티콘, 인사 → NO
```

비용: 메시지당 약 $0.001~0.002

### 4.2 2단계: 파싱 (Claude Sonnet)

목적: 주문 메시지에서 구조화된 정보 추출

입력 예시:
```
화주김: 역삼동에서 해운대 박스3개 전자제품 35만 오후3시까지
```

출력:
```json
{
  "sender": "화주김",
  "parsed": {
    "origin": "서울 강남구 역삼동",
    "destination": "부산 해운대구",
    "cargo": "박스 3개 (전자제품)",
    "deadline": "오늘 오후 3시",
    "requestedPrice": 350000,
    "vehicleType": null,
    "specialNotes": null
  },
  "confidence": 0.92
}
```

비용: 메시지당 약 $0.01~0.03

### 4.3 이미지 분류 기준

| 이미지 유형 | 분류 | 처리 |
|------------|------|------|
| 운송장/송장 사진 | YES | OCR + 파싱 |
| 물품 사진 (배송할 짐) | YES | 물품 설명 추출 |
| 지도 캡처 (위치 표시) | YES | 주소 추출 |
| 밈/이모티콘/일상 사진 | NO | 무시 |

### 4.4 비용 비교

| 방식 | 하루 100건 기준 |
|------|-----------------|
| 전부 정밀 파싱 | 100건 x $0.02 = $2.00/일 |
| **2단계 필터링** | 100건 x $0.002 + 5건 x $0.02 = **$0.30/일** |

## 7. 배차앱 연동

### 7.1 Adapter 패턴

```
                Dispatch Adapter Layer
    +------------+  +------------+  +------------+
    | 24시전국특송 |  | 인성퀵      |  | 기타 앱     |
    | Adapter    |  | Adapter    |  | Adapter    |
    +------+-----+  +------+-----+  +------+-----+
           |               |               |
           v               v               v
       API 호출        웹 자동화         API 호출
```

### 7.2 연동 방식

| 방식 | 설명 | 적용 상황 |
|------|------|-----------|
| 공식 API | REST API로 직접 전송 | 배차앱이 API를 제공하는 경우 |
| 웹 자동화 | Playwright로 웹 페이지 입력 자동화 | API 없이 웹만 있는 경우 |

새 배차앱 추가 시 해당 Adapter만 개발하면 됨.

### 7.3 에러 처리

```
전송 시도 → 성공 → DB에 DISPATCHED 기록
         → 실패 → 3회 재시도 → 성공 → 기록
                            → 최종 실패 → 사용자 알림
                                        → FAILED 상태, 수동 처리 안내
```

## 8. 기술적 리스크 및 대응

| 리스크 | 영향 | 대응 방안 |
|--------|------|-----------|
| 카톡 PC 앱 업데이트로 UI 구조 변경 | Monitor 동작 불가 | UI 요소 탐색 로직을 설정 파일로 분리, 빠른 대응 가능 구조 |
| AI 파싱 정확도 부족 | 잘못된 주문 정보 전송 | 사용자 확인 단계 필수, 신뢰도 낮은 건 경고 표시 |
| 20개+ 방 순회 시 딜레이 | 메시지 캡처 지연 (~60초) | 우선순위 방 설정, 중요 방 먼저 순회 |
| 카톡 창 포커스 충돌 | 모니터링 전용 PC 필요 | 별도 PC 또는 VM에서 운영 권장 |
| 배차앱 API 변경/미제공 | 연동 불가 | Adapter 패턴으로 격리, 웹 자동화 fallback |

## 9. 향후 확장 고려사항

- **모바일 지원**: Android Accessibility Service를 활용한 모바일 카톡 모니터링
- **AI 학습**: 반복 수정 데이터를 활용한 파싱 정확도 개선
- **다중 사용자**: 서버 기반 아키텍처로 전환 시 여러 직원이 동시 사용 가능
- **통계/분석**: 주문량, 경로별 통계, 화주별 이용 패턴 분석
